# Skill: `rethink`（协作式反思，`/rethink` 命令）

> **Alpha 说明**：Alpha 版本将本设计规范整合为 `skills/rethink/SKILL.md`（[vercel-labs/skills](https://github.com/vercel-labs/skills) 格式），用户通过 `npx skills add` 安装，自动适配各家 Agent。

## 一、设计理念

有时候用户对 Agent 的工作结果不满意，但说不清楚哪里不对。这种"模糊的不满"是一个常见而真实的困境：

- 用户知道"不够好"，但不知道"好"是什么样子
- 直接说"重做"可能导致 Agent 重复同样的方向
- 使用 PUA 话术（"你这算什么水平？"）既无效又有害
- 用户需要的不是 Agent 更努力，而是 Agent 帮他一起想清楚

`/rethink` 就是为这个场景设计的：**Agent 和用户一起反思，一起找到方向。**

## 二、三阶段协作流程

```
用户输入: /rethink
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│  第一阶段：Agent 自我审视（立即执行）                       │
│                                                         │
│  Agent 诚实地审视自己刚才的工作：                          │
│  * 我采取了什么方法？为什么选择这个方向？                   │
│  * 我做了哪些假设？这些假设有没有问题？                    │
│  * 有哪些替代方案我没有尝试？                              │
│  * 我的输出可能在哪些方面不够好？                          │
│                                                         │
│  输出：坦诚的自我评估，指出具体的薄弱点                    │
│  （这本身就是在示范知性诚实——PUA 的反面）                  │
└──────────────────────┬──────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────┐
│  第二阶段：引导用户反思（协作探索）                         │
│                                                         │
│  Agent 用结构化的问题帮用户梳理模糊的不满：                 │
│                                                         │
│  1. "我的自我评估和你的感觉一致吗？                        │
│      还是问题出在完全不同的地方？"                         │
│                                                         │
│  2. "哪个方面最让你觉得'不对劲'？"                        │
│     □ 方向/思路不对                                      │
│     □ 结果太浅/不完整                                    │
│     □ 风格/格式不符合需要                                │
│     □ 其他我没有想到的方面                                │
│                                                         │
│  3. "你心中'好的结果'大概是什么样的？                      │
│      哪怕只是一个感觉、一个类比、一个参考？"               │
│                                                         │
│  4. "有没有我遗漏的背景信息——                             │
│      约束条件、偏好、目标？"                               │
└──────────────────────┬──────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────┐
│  第三阶段：综合与行动计划                                  │
│                                                         │
│  Agent 综合自我反思和用户反馈：                            │
│  * "根据我的审视和你的反馈，核心问题似乎是 [X]。"          │
│  * "我的修正方案是：[具体计划]"                            │
│  * "这个方向对吗？确认后我开始执行。"                      │
└─────────────────────────────────────────────────────────┘
```

## 三、使用示例

**场景**：用户让 Agent 重构一段代码，结果出来了但用户说不上来哪里不满意。

```
用户: /rethink

Agent（第一阶段 — 自我审视）:
────────────────────────────────
让我审视一下刚才的重构。

我的方法是把大函数拆分成多个小函数，每个负责一个步骤。
这在结构上更清晰了，但我现在反思，有几个可能的问题：

1. 我保持了原来的命令式风格，没有考虑是否用声明式/管道式
   写法会更好
2. 拆分出的函数命名偏向"做了什么"（processData）而非
   "为什么做"（validateUserInput）
3. 我没有检查这些函数是否有已存在的工具库可以替代

你的不满是和这些有关，还是完全不同的方向？

用户: 嗯...第一点有点关系。我感觉代码虽然拆开了但还是
      读起来很"碎"，不知道怎么说得更清楚

Agent（第二阶段 — 引导反思）:
────────────────────────────────
"碎"是一个很好的感觉词。让我帮你把它翻译成具体的技术需求：

"碎"可能意味着：
a) 函数太多太小，跳来跳去反而不如原来一眼看完
b) 抽象层次不统一——有的函数很高层，有的很细节
c) 想要的不是"拆分"，而是"用更优雅的方式表达同一个逻辑"
   （比如用 pipeline/chain 风格）

哪个最接近你的感觉？

用户: c。我想要的是那种一行行读下来就能懂的感觉，
      不是跳来跳去看函数定义

Agent（第三阶段 — 综合与行动）:
────────────────────────────────
明白了。核心需求不是"拆分函数"，而是"提高可读性"。

我的修正方案：
* 用 pipeline 风格重写，让数据流从上到下一目了然
* 减少独立函数，改用链式调用 (map/filter/reduce)
* 保留关键步骤的注释，但让代码本身讲述逻辑

这个方向对吗？确认后我重新来过。
```

## 四、扩展命令

```bash
# NVC 翻译器：把任何一句话翻译成 NVC 版本（引用 nvc Skill 的翻译引擎）
/rethink translate "话术"

# 分析一段对话中的 PUA 模式
/rethink analyze "粘贴对话内容"

# Agent 沟通模式自检
/rethink selfcheck
```
